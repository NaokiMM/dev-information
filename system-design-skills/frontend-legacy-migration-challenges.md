今回の開発では、Vue2・Vue3・CakePHPが混在する環境でフロントエンド開発を行いました。
特に課題だったのは、同じ機能でも実装方法がバラバラだった点です。

例えば、Vue2ではOptions APIで書かれている一方、Vue3ではComposition APIが使われていました。

// Vue2（Options API）
export default {
  data() {
    return {
      count: 0
    }
  },
  methods: {
    increment() {
      this.count++
    }
  }
}

// Vue3（Composition API）
import { ref } from 'vue'

export default {
  setup() {
    const count = ref(0)
    const increment = () => {
      count.value++
    }
    return { count, increment }
  }
}

このように設計思想が異なるため、メンバーごとに実装方針が分かれ、保守性が低下していました。

さらに、CakePHP側ではテンプレートに直接JSが埋め込まれているケースもあり、責務が曖昧でした。

<!-- CakePHPテンプレート -->
<button onclick="increment()">+</button>

この状態では、フロントとバックエンドの境界が不明確になり、修正時の影響範囲が読みづらくなります。

そこで私は、Vue3ベースに統一する方針を提案し、
コンポーネント設計と状態管理のルールを簡易的に整理しました。

この経験から、技術スタックの統一とアーキテクチャ設計の重要性、
そして移行期におけるルール整備の必要性を強く学びました。
